# ADR 0005: Keep Authored Alteran Source in `src/`, Not in `.runtime/`

## Status

Accepted

## Context

Alteran originally allowed an uncomfortable ambiguity between:

- authored Alteran source code
- materialized runtime files under `.runtime/`

That ambiguity is risky because `.runtime/` is intentionally a managed area:

- it is cleaned by maintenance commands
- it is regenerated by bootstrap flows
- it is naturally treated as disposable by users and by `.gitignore`

Keeping the only authoritative Alteran implementation inside `.runtime/` would
make it too easy to delete real source code while believing only generated
artifacts were being removed.

Alteran also needs a clean separation between:

- authored repository source
- generated runtime material
- prepared publication outputs

## Decision

Alteran keeps its authored source-of-truth in repository `src/`:

- `src/alteran/`
- `src/tools/`
- `src/libs/`

The project-local `.runtime/` tree is materialized/generated output, not the
canonical home of authored source.

More specifically:

1. Repository-level `alteran.ts` delegates to `src/alteran/mod.ts`.
2. `refresh`, `init`, and `ensure-env` materialize `.runtime/alteran`,
   `.runtime/tools`, and `.runtime/libs` from authored source when local source
   is available.
3. `prepare_jsr` and other release tooling should package from `src/`, not from
   `.runtime/`.
4. `.runtime/` should be safe to ignore in git and safe to reconstruct from
   source/bootstrap inputs.
5. Bootstrap and env script generators belong to Alteran-owned template modules
   under `src/alteran/templates/`.

## Consequences

Positive:

- source-of-truth is explicit and harder to destroy by accident
- `.runtime/` can honestly behave as a managed/generated area
- publication tooling and repository development use the same authored source
- runtime materialization becomes a clear projection from source to generated
  runtime

Tradeoffs:

- Alteran must maintain materialization logic from `src/` into `.runtime/`
- repository bootstrap must distinguish repository source mode from ordinary
  managed-project mode

## Rejected Alternatives

### Keep authored Alteran implementation directly in `.runtime/`

Rejected because it confuses generated/runtime state with real source code and
makes cleanup flows too risky.

### Store bootstrap templates as neighboring shell files outside Alteran source

Rejected because template ownership then drifts away from the TypeScript code
that regenerates and reasons about those files.
